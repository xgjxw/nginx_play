# Nginx ngx_buf_t 缓冲区结构使用指南

## 何时使用

- 在编写nginx C语言模块时，需要读取或写入内存/文件数据
- 需要构造或解析用于I/O链（如输出过滤器链）的缓冲区
- 遇到涉及`ngx_buf_t`结构体的内存管理或数据传输逻辑

## 核心操作流程

1. **理解缓冲区用途**：`ngx_buf_t`用于保存待写入目标或从源读取的数据，可同时引用内存块、文件区域，或两者兼有。
2. **分配结构体**：`ngx_buf_t`结构本身通常从内存池分配，但其指向的数据缓冲区需独立分配。
3. **设置关键字段**：根据数据来源（内存或文件）及用途，正确设置以下字段：
   - `start` / `end`：分配的内存块边界
   - `pos` / `last`：实际使用的内存范围（必须满足 `start ≤ pos ≤ last ≤ end`）
   - `file_pos` / `file_last`：文件数据的偏移范围（单位：字节）
   - `file`：指向`ngx_file_t`对象的指针（若引用文件）
4. **设置标志位**：根据缓冲区特性启用相应标志：
   - `temporary`：缓冲区内容可写
   - `memory`：缓冲区内容只读（如字符串常量）
   - `in_file`：包含文件数据
   - `flush`：要求刷新此前所有缓冲区
   - `last_buf`：表示整个响应的最后一个缓冲区
   - `last_in_chain`：表示当前请求/子请求无后续缓冲区
   - `recycled`：缓冲区可被重用，应尽快消费
   - `sync`：无实际数据，仅作同步信号
   - `temp_file`：数据位于临时文件中
5. **使用tag与shadow**：
   - `tag`：用于标识缓冲区来源，便于重用管理
   - `shadow` / `last_shadow`：用于关联原始缓冲区（如压缩后数据引用原始输入）

## 注意事项

- 确保指针有效性：`pos`、`last`等指针必须在`[start, end]`范围内
- 文件与内存缓冲区可共存，但需通过标志位明确区分
- 不适用于非nginx环境或非C语言实现

完成配置后，返回的`ngx_buf_t`实例可用于nginx I/O链处理。
