# Nginx SCGI 上游容错与重试配置

## 何时使用
- 已配置多个 SCGI 服务器组成 server group
- 需要提升 SCGI 后端服务的可用性与容错能力
- 出现连接错误、超时或特定 HTTP 状态码（如 500、503、429）时希望自动重试
- 确保仅在安全条件下（如未开始响应传输）触发重试

## 核心操作步骤

### 1. 配置重试触发条件
使用 `scgi_next_upstream` 指令指定哪些失败类型应触发重试：
- `error`：连接、发送或读取响应头时出错（始终视为失败）
- `timeout`：连接、发送或读取超时（始终视为失败）
- `invalid_header`：收到空或无效响应头（始终视为失败）
- `http_500` / `http_503` / `http_429`：仅当显式列出时才视为失败
- `non_idempotent`：允许对 POST 等非幂等请求重试（Nginx ≥1.9.13）
- `off`：完全禁用重试

> 注意：`http_403` 和 `http_404` 永不视为失败，即使在指令中列出。

### 2. 设置重试限制
- **最大尝试次数**：通过 `scgi_next_upstream_tries number` 设置（默认 0 表示无限制）
- **总重试时间上限**：通过 `scgi_next_upstream_timeout time` 设置（默认 0 表示无限制）

### 3. 验证重试可行性
重试仅在以下条件下执行：
- 尚未向客户端发送任何响应数据
- 若已开始流式响应传输，则无法重试

### 4. 调整相关超时参数
- `scgi_connect_timeout`：建立连接的超时时间（默认 60 秒，建议 ≤75 秒）
- `scgi_send_timeout`：两次写操作之间的超时（默认 60 秒）
- `scgi_read_timeout`：两次读操作之间的超时（默认 60 秒）

## 执行结果
根据失败类型、重试配置及约束条件，决定是否将请求转发至下一可用 SCGI 服务器，从而实现容错与高可用。
